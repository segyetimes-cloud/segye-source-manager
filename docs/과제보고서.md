## 1. 과제 개요

### 1.1 배경

언론사 취재 활동의 핵심 자산은 **취재원(Source)** 이다. 기자 개인이 수년간 쌓아온 취재원 네트워크는 단순한 연락처 목록을 넘어, 정보의 신뢰도·접근 가능성·관계망을 포함하는 복합 데이터다. 그러나 현실에서 이 소중한 자산은 기자의 개인 스마트폰 주소록과 메모장에 분산되어 있어, 아래의 구조적 문제를 안고 있다.

- 기자 이직·퇴직 시 취재원 정보가 조직에서 사라짐
- 부서 내 정보 공유 불가 → 중복 취재·중복 접촉 발생
- 취재원 인맥 파악 불가 → 탐사 기획 시 연결 고리 놓침
- 민감 정보(정치적 성향, 오프더레코드 발언 등)의 무단 유출 위험

### 1.2 과제 목표

| 목표 | 내용 |
|------|------|
| 취재원 DB 구축 | 전사 공유 가능한 구조적 취재원 데이터베이스 |
| 접근 제어 | 역할별 차등 열람 권한으로 민감 정보 보호 |
| 네트워크 시각화 | 취재원 간 인맥 지도 자동 생성 |
| 협업 기능 | 기자 간 정보 공유·정보보고 시스템 |
| 감사 추적 | 모든 열람·수정 이력 자동 기록 |

### 1.3 개발 범위

- **프론트엔드** : Next.js 16 App Router (TypeScript)
- **백엔드/DB** : Supabase (PostgreSQL + Row Level Security)
- **배포 환경** : Vercel (서버리스)
- **개발 기간** : 약 8주

## 2. 기존 업무 방식의 문제점

### 2.1 분산된 취재원 관리

```
기자 A: 스마트폰 연락처 + 카카오톡 메모
기자 B: 구글 스프레드시트 (개인 계정)
기자 C: 노션 개인 워크스페이스
데스크 : 수기 수첩
```

**결과**: 같은 취재원에게 여러 기자가 별도 접촉 → 취재원 피로 유발, 조직 신뢰 훼손

### 2.2 퇴사 시 정보 소멸

기자 1인 평균 보유 취재원 수 **200~500명** 추정. 퇴사 시 이 정보는 개인 디바이스와 함께 사라진다. 언론사는 수년간 축적된 네트워크 자산을 순간에 잃는다.

### 2.3 민감 정보 보호 불가

엑셀·노션에 저장된 취재원 성향, 비공개 제보 내용은 권한 구분 없이 파일 공유만으로 유출 가능하다. 실제로 취재원 신상 정보 유출 사고는 업계 전반에서 반복되고 있다.

### 2.4 관계망 파악 불가

기자 개인의 머릿속에만 존재하는 취재원 인맥 정보는 탐사 보도·특종 기획 시 공유될 수 없어 협업의 큰 장벽이 된다.

### 2.5 문제 요약표

| 문제 | 발생 빈도 | 영향 |
|------|----------|------|
| 취재원 중복 접촉 | 주 1~2회 이상 추정 | 취재원 신뢰 저하 |
| 퇴사 시 정보 소멸 | 인원 변동 시마다 | 조직 자산 손실 |
| 민감 정보 무단 공유 | 간헐적 | 법적 리스크, 취재원 안전 위협 |
| 인맥 지도 부재 | 상시 | 기획 취재 비효율 |
| 연락 이력 미기록 | 상시 | 후속 취재 단절 |

## 3. 시스템 설계 방향

### 3.1 설계 원칙

1. **Zero-Trust 정보 보호** : 모든 접근은 인증·인가 후 허용, 최소 권한 원칙
2. **계층적 권한 구조** : 역할에 따라 열람 범위를 세밀하게 제어
3. **감사 불가능성 제거** : 모든 행위는 자동으로 로그에 기록
4. **기자 친화적 UX** : 빠른 등록, 직관적 인터페이스

### 3.2 역할 계층 구조

```
슈퍼관리자 (superadmin)
    └── 편집인 (publisher)
         └── 국장 (editor)
              └── 부국장 (section_editor)
                   └── 부장 (admin) ← 소속 부서만 관할
                        └── 차장 (deputy)
                             └── 기자 (reporter)
```

### 3.3 민감 정보 접근 제어 매트릭스

| 정보 구분 | 기자 | 차장 | 부장 | 부국장 이상 |
|-----------|------|------|------|------------|
| 기본 정보 | ✅ | ✅ | ✅ | ✅ |
| 공개 노트 | ✅ | ✅ | ✅ | ✅ |
| 민감 노트(내 것) | ✅ | ✅ | ✅ | ✅ |
| 민감 노트(타인) | ❌ | ✅ | ✅ | ✅ |
| 민감 정보(personal_notes) | ❌→승인 | ✅ | ✅ | ✅ |
| 정보보고(전체) | ❌ | ❌ | 소속부서 | ✅ |

## 4. 주요 기능 상세

### 4.1 취재원 등록 및 관리

**기본 등록 항목**
- 이름, 소속, 직책, 부서
- 연락처(전화/이메일 주/보조)
- 생년월일, 출신지역
- 학력 정보(고교·대학·대학원·전공)
- 고시/기수
- SNS 링크

**취재 관련 항목**
- **온더레코드 동의 여부** : ✅온더레코드 / 🟡백그라운드 / 🔴오프더레코드
- 공개 정보(편집국 전원 열람)
- 민감 정보(차장 이상 또는 승인자만 열람)
- 태그, 전문 분야

**완성도 점수 자동 산출**
```
기본정보 20점 + 연락처 20점 + 학력 20점 + 정보 노트 40점 = 100점
```

### 4.2 정보 노트 시스템 (Source Notes)

여러 기자가 한 취재원에 대해 공유 노트를 작성하는 기능. 기존 메신저 공유 방식을 대체한다.

- **공개 노트** : 모든 기자 열람 가능
- **민감 노트** : 작성자 본인 + 차장 이상만 열람
- 포인트 시스템 연동 (+10pt/건)
- 중복 문장 자동 감지 통합 뷰 제공

### 4.3 연락 이력 관리 (Contact Logs)

```
연락 방법 : 전화 / 문자 / 이메일 / 대면 / 기타
연락 일시 : datetime 기록
내용 요약 : 자유 기록
결과 : 인터뷰 수락, 재연락 예정 등
다음 팔로업 날짜 : 팔로업 배너로 강조 표시
```

가장 가까운 팔로업 일정을 카드 상단에 배너로 자동 표시. 3일 이내 팔로업은 녹색 강조.

### 4.4 직책 이력 자동 관리

소속·직책 변경 시 자동으로 이력에 쌓이는 타임라인 구조. 이직 추적, 경력 파악에 활용.

### 4.5 정보보고 시스템 (Information Reports)

취재 과정에서 수집한 정보를 구조화된 보고서로 작성·공유하는 기능.

**가시성 설정**
| 레벨 | 대상 |
|------|------|
| 전체 공개 | 편집국 전원 |
| 데스크 이상 | 부장급 이상 |
| 팀 공개 | 소속 부서 |
| 작성자만 | 본인 한정 |

**카테고리** : 단독 / 인터뷰 / 기획 / 일반

### 4.6 네트워크 그래프

취재원 간 연결 관계를 실시간 인터랙티브 그래프로 시각화.

**자동 연결 기준**
- 동일 소속 기관 (동료)
- 대학 동문, 고교 동문
- 고시·시험 동기
- 동향 (출신 지역)
- 직접 언급 관계
- 수동 등록 관계

**검색 기능**
- 1촌 검색 : 직접 일치 취재원 강조
- **2촌 검색** : 일치 취재원 + 그 연결 취재원까지 확장 강조 (신규)

### 4.7 포인트 시스템

기자의 시스템 기여도를 포인트로 정량화하여 동기 부여.

| 행위 | 포인트 |
|------|--------|
| 취재원 등록 (기본) | +5~30pt |
| 정보 노트 추가 | +10pt |
| 취재원 활용 | +5pt |
| 정보보고 우수상 | 관리자 수여 |
| 도움 제공/수락 | +포인트 |

### 4.8 승인 워크플로

민감 정보 취재원 열람 신청 → 데스크 승인 → 30일 기한부 접근 허용.

## 5. 도입 전후 비교

### 5.1 업무 프로세스 비교

| 구분 | 도입 전 | 도입 후 |
|------|---------|---------|
| **취재원 등록** | 개인 주소록/메모앱 | 전사 통합 DB에 구조화 등록 |
| **정보 검색** | 동료에게 직접 물어봄 | 검색어 1초 이내 결과 |
| **관계 파악** | 기자 머릿속에만 존재 | 인터랙티브 네트워크 그래프 |
| **연락 이력** | 카카오톡 채팅 기록에 묻힘 | 타임라인으로 체계적 기록 |
| **정보 공유** | 메신저/이메일로 비구조 공유 | 정보보고 시스템으로 구조화 |
| **민감 정보 보호** | 파일 공유 시 무방비 노출 | 역할 기반 접근 제어 |
| **퇴사자 정보** | 개인 디바이스와 함께 소멸 | 조직 자산으로 영구 보존 |
| **감사 추적** | 불가능 | 모든 열람/수정 자동 기록 |
| **인맥 탐색** | 부서 내 구두 문의 | 2촌 네트워크 검색 |
| **팔로업 관리** | 개인 캘린더 또는 기억 의존 | 팔로업 배너 자동 알림 |

### 5.2 보안 수준 비교

| 항목 | 도입 전 | 도입 후 |
|------|---------|---------|
| 접근 통제 | 없음 (파일 공유 기반) | JWT 인증 + 역할 기반 RLS |
| 민감 정보 분류 | 없음 | 공개/민감 2단계 분류 |
| 정보 유출 추적 | 불가 | 워터마크 + 감사 로그 |
| 취재원 신원 보호 | 없음 | 열람 승인 워크플로 |
| 복사 감지 | 불가 | 제로-폭 워터마크 자동 삽입 |

### 5.3 협업 효율 비교

```
[도입 전]
기자 A가 취재원 X 접촉
        ↓ 보고 없음
기자 B도 같은 취재원 X 독립 접촉
        ↓ 취재원 피로, 혼선
취재원 X 언론 기피 시작

[도입 후]
기자 A가 취재원 X 등록 + 연락 이력 기록
        ↓ 시스템 알림
기자 B가 시스템에서 기자 A의 이력 확인 후 협의
        ↓ 중복 접촉 방지
취재원 X 신뢰 유지
```

## 6. 기술 구현 내용

### 6.1 기술 스택

| 영역 | 기술 | 선택 이유 |
|------|------|----------|
| 프레임워크 | Next.js 16 App Router | SSR/SSG, API Routes 통합, 최신 React 패턴 |
| 언어 | TypeScript | 타입 안전성, 유지보수성 |
| 데이터베이스 | Supabase (PostgreSQL) | RLS 내장, 실시간, 인증 통합 |
| 스타일 | Inline styles (CSS-in-JS) | 서버 컴포넌트 호환, 런타임 zero |
| 그래프 | react-force-graph-2d | D3 기반 WebGL 가속 네트워크 시각화 |
| 배포 | Vercel | Edge 최적화, 자동 CI/CD |

### 6.2 데이터베이스 주요 테이블

```sql
sources              -- 취재원 기본 정보
source_positions     -- 직책 이력 타임라인
source_notes         -- 다중 기자 공유 노트
contact_logs         -- 연락 이력 + 팔로업
source_relationships -- 취재원 간 관계망
source_access_approvals -- 민감 정보 열람 승인
information_reports  -- 정보보고
report_sources       -- 보고서-취재원 연결
point_transactions   -- 포인트 거래 내역
audit_logs           -- 전체 감사 로그
```

### 6.3 보안 구현 핵심 기술

**Row Level Security (RLS)**
```sql
-- 예시: 개인 목록은 소유자 또는 관리자만
CREATE POLICY "sources_select" ON sources
  FOR SELECT USING (
    visibility = 'shared'
    OR owner_id = auth.uid()
    OR current_user_role() IN ('admin','superadmin',...)
  );
```

**제로-폭 워터마크**
```
복사된 텍스트에 사용자 ID를 U+200B/U+200C 비트로 인코딩
→ 육안 식별 불가, 디지털 포렌식으로 유출자 추적 가능
```

## 7. 보안 및 데이터 보호 설계

### 7.1 다층 보안 아키텍처

```
Layer 1: Supabase Auth (JWT 토큰 인증)
Layer 2: Next.js 미들웨어 (서버사이드 세션 검증)
Layer 3: Row Level Security (DB 레벨 행 단위 접근 제어)
Layer 4: API Route 역할 검증 (비즈니스 로직 레벨)
Layer 5: UI 레벨 조건부 렌더링 (표시 제어)
```

### 7.2 감사 로그 항목

| 이벤트 | 기록 내용 |
|--------|----------|
| 열람 | 사용자, IP, User-Agent, 취재원 ID, 시각 |
| 수정 | 변경 전/후 값, 편집자, 시각 |
| 민감 정보 열람 | 별도 `view_private` 액션으로 분리 기록 |
| 내보내기 | 건수, 워터마크 토큰, 필터 조건 |
| 승인/거절 | 승인자, 사유, 만료일 |

### 7.3 정보 유출 방지 장치

1. **워터마크 인코딩** : 복사 시 사용자 식별 코드 자동 삽입
2. **민감 정보 마스킹** : 권한 없는 필드는 서버에서 null로 응답
3. **열람 승인 만료** : 30일 후 자동 접근 취소
4. **VPN 접속 감지** : IP 기반 VPN 여부 플래그 기록
5. **개인 목록 격리** : 소유자 외 접근 시 404 반환 (정보 존재 자체 노출 방지)

### 8.1 업무 효율성

- **히스토리 자산** : 10년치 접촉 이력이 검색 가능한 형태로 보존

- **민감 정보 노출 사고** : 구조적으로 차단
- **내부자 위협** : 감사 로그 + 워터마크로 억제
- **GDPR/개인정보 컴플라이언스** : 접근 이력 추적 가능

### 8.4 협업 문화 변화
